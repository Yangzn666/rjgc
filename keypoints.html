软件工程导论重点知识汇总

★基本概念
  • 软件危机：软件开发和维护过程中遇到的一系列严重问题
    • 例子：IBM的System/360操作系统项目，预算1000万美元，最终花费超过5000万美元
    • 例子：早期银行系统频繁出现计算错误，导致客户资金错误
    • 表现：成本超支、进度延误、质量低下、维护困难
    • 类比：像建造房屋时没有设计图纸，工人边建边改，结果工期拖延、成本超支、质量低劣
    软件危机关系图：
    需求不明确
    ↓
    设计不当 → 代码质量差
    ↑ ↗
    工具落后 ← 测试不足
  • 软件工程：应用计算机科学理论和技术以及工程管理原则和方法的学科
    • 例子：像建筑工程一样，需要系统化的方法和管理
    • 目标：提高软件质量、降低开发成本、控制开发进度
  • 软件生命周期：软件定义、开发、维护的全过程
    • 例子：从需求分析到最终退役的全过程，如Office软件的整个生命周期
    • 类比：一个人从出生、成长、工作到退休的整个过程
    软件生命周期阶段图：
    定义阶段：[可行性研究] → [需求分析]
    ↓
    开发阶段：[总体设计] → [详细设计] → [编码实现] → [测试]
    ↓
    维护阶段：[软件维护]
  • 软件过程：为获得高质量软件所需完成的一系列任务的框架
    • 例子：瀑布模型、敏捷开发等开发流程
    • 类比：施工流程，包括设计、打地基、建主体、装修等步骤
    软件过程模型对比：
    瀑布模型： 线性顺序
    [需求]→[设计]→[编码]→[测试]→[维护]

    螺旋模型： 风险驱动
    [风险分析] → [原型开发] → [评估] → [规划]
    ↑ ↓
    ←———————————————迭代———————————————————————
★软件工程三要素
  • 方法：软件开发的方法论
    • 例子：结构化分析设计方法（如：传统银行系统开发）
    • 例子：面向对象分析设计方法（如：现代Web应用开发）
    • 例子：敏捷开发方法（如：创业公司快速迭代产品）
    • 类比：就像建筑中的施工方法（钢筋混凝土结构 vs 木结构 vs 钢结构）
  • 工具：支持软件开发的工具
    • 需求分析工具：Visio、Enterprise Architect
    • 编码工具：Visual Studio、IntelliJ IDEA、Eclipse
    • 测试工具：JUnit、Selenium、LoadRunner
    • 版本控制工具：Git、SVN
    • 类比：就像建筑中的工具（锤子、锯子、起重机、测量仪器）
  • 过程：软件开发的步骤和流程
    • 瀑布模型：需求分析→系统设计→编码实现→测试→维护
    • 敏捷开发：迭代开发，持续交付
    • DevOps：开发运维一体化
    • 类比：就像建筑中的施工流程（设计→打地基→建框架→装修→验收）

    [工具] ←—— 支持 ————————

    三要素协同作用：
    方法论指导开发过程，工具支撑方法实施
★软件生命周期阶段
  • 定义阶段：可行性研究、需求分析
  • 开发阶段：总体设计、详细设计、编码实现、测试
  • 维护阶段：软件维护
★软件维护类型
  • 改正性维护：诊断和改正软件错误
  • 适应性维护：适应环境变化
  • 完善性维护：增加功能或改进性能（占比最多）
  • 预防性维护：为未来可能的改变做准备
★模块独立性
  • 目标：高内聚、低耦合
    • 高内聚例子：'用户登录'模块只包含验证用户名、密码和创建会话的功能
    • 低耦合例子：计算器模块只依赖输入模块提供数字，不依赖其他模块
    • 坏例子：用户管理模块既处理登录、又处理订单、还处理日志（低内聚）；模块A直接修改模块B的内部变量（高耦合）
    • 类比：乐高积木，模块间连接简单（低耦合），每个积木块功能单一（高内聚）
    模块独立性目标：
    高内聚：┌─────────────────┐
    │ 用户登录模块 │
    │ - 验证用户名 │
    │ - 验证密码 │
    │ - 创建会话 │
    └─────────────────┘
    低耦合：模块间依赖关系简单
  • 内聚类型（由高到低）：功能内聚、顺序内聚、通信内聚、过程内聚、时间内聚、逻辑内聚、偶然内聚
    • 功能内聚：模块完成单一功能
      • 例子：计算圆面积模块只负责计算圆面积
      • 优点：模块功能明确，易理解、易维护
    • 顺序内聚：数据依次处理
      • 例子：读取数据→处理数据→输出结果
      • 优点：数据流清晰，处理有序
    • 偶然内聚：模块元素无关联
      • 例子：工具箱模块包含排序、打印、网络连接等多个无关功能
      • 缺点：模块混乱，难以维护
      内聚类型对比：
      高 → 低
      功能内聚：[单一任务] ✓
      顺序内聚：[数据流处理] ✓
      通信内聚：[共享数据] △
      ...
      偶然内聚：[无关联功能] ✗
  • 耦合类型（由低到高）：非直接耦合、数据耦合、标记耦合、控制耦合、公共耦合、内容耦合
    • 数据耦合：通过参数传递数据
      • 例子：函数调用时传递基本数据类型或简单结构
      • 如：calculateArea(radius) // 只传递半径值
    • 控制耦合：传递控制信号
      • 例子：传递标志参数控制另一个模块的行为
      • 如：processData(data, flag) // flag决定处理方式
    • 内容耦合：一个模块直接访问另一模块内部
      • 例子：模块A直接修改模块B的内部变量
      • 后果：模块紧密绑定，修改一个会影响另一个
      耦合类型对比：
      低 → 高
      非直接耦合：无依赖 [最佳] ✓
      数据耦合：参数传递 [较好] △
      控制耦合：控制信号 [一般] ○
      内容耦合：直接访问内部 [最差] ✗
★测试方法
  • 黑盒测试：不考虑内部结构，测试功能
    • 例子：测试登录功能，只关注输入用户名密码后是否能成功登录
    • 例子：测试计算器，只验证输入2+3是否得到5，不关心内部如何计算
    • 类比：测试微波炉，只看是否能加热食物，不看内部电路如何工作
    • 关注点：输入-输出的正确性
    黑盒测试示意图：
    输入 → [系统] → 输出
    (不关心内部)
  • 白盒测试：考虑内部结构，测试逻辑路径
    • 例子：测试一个判断成绩等级的函数，需要测试if-else的所有分支路径
    • 例子：测试循环结构，确保循环边界条件正确
    • 类比：检查汽车发动机，需要打开引擎盖看内部零件是否正常
    • 关注点：代码的逻辑结构和执行路径
    白盒测试示意图：
    输入 → [if条件 → 循环 → 逻辑路径] → 输出
    ↑
    关注内部实现
  • 测试步骤：单元测试 → 集成测试 → 确认测试 → 系统测试
    • 单元测试：测试单个模块或函数
      • 例子：测试一个排序函数是否能正确排序数组
      • 开发者负责，通常使用JUnit等框架
    • 集成测试：测试模块间接口和交互
      • 例子：测试用户模块和订单模块之间的数据传递
      • 确保模块组合后功能正确
    • 确认测试：验证软件是否满足需求
      • 例子：验证登录功能是否符合用户需求规格说明书
      • 也称为验收测试
    • 系统测试：测试整个系统
      • 例子：在真实环境下测试整个电商系统的所有功能
      • 验证系统在实际环境中的表现
      测试步骤层次图：
      系统测试 ────────────────────────────── [整个系统]
      ↓
      集成测试 ─────── [模块间接口] ─────── [模块组合]
      ↓
      单元测试 ── [单个模块/函数] ── [最小测试单位]

      确认测试 ────────────────── [需求验证] ── [用户视角]
