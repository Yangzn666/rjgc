★软件危机

定义：软件开发过程中遇到的各种问题和困难

表现：

  • 软件成本高：如IBM的System/360操作系统项目，预算1000万美元，最终花费超过5000万美元
  • 软件质量差：程序存在大量错误，可靠性差，如早期的银行系统经常出现计算错误
  • 开发进度难以控制：项目延期严重，如原定6个月完成的项目可能拖到2-3年
  • 软件维护困难：代码难以理解，修改容易引入新错误

原因：

  • 需求不明确：用户需求频繁变更，开发过程中不断提出新要求
  • 开发技术落后：缺乏有效的开发方法和工具
  • 管理困难：项目缺乏有效计划和监控机制
  • 缺乏正确的方法指导：没有规范的开发流程

生活化类比：软件危机就像建造房屋时没有设计图纸，工人边建边改，结果工期拖延、成本超支、质量低劣。

★软件工程基本原理

理解其核心思想和实践意义（共7条原理）

软件工程的7条基本原理：

  1. 用分阶段的生命周期计划严格管理
  2. 坚持进行阶段评审
  3. 实行严格的产品控制
  4. 采用现代程序设计技术
  5. 结果应能清楚地审查
  6. 开发小组人员应少而精
  7. 承认不断改进软件工程实践的必要性
★软件工程三要素

方法、工具、过程

  • 方法：软件开发的方法论
    • 例子：结构化分析设计方法、面向对象分析设计方法
    • 类比：就像建筑中的施工方法（钢筋混凝土结构 vs 木结构）
  • 工具：支持软件开发的工具
    • 例子：需求分析工具（如Visio）、编码工具（如IDE）、测试工具（如JUnit）
    • 类比：就像建筑中的工具（锤子、锯子、起重机）
  • 过程：软件开发的步骤和流程
    • 例子：瀑布模型、敏捷开发流程
    • 类比：就像建筑中的施工流程（设计→打地基→建框架→装修）
★软件工程方法

传统软件工程方法 vs 面向对象软件工程方法

传统软件工程方法：

  • 以功能分解为核心
  • 自顶向下设计
  • 结构化分析和设计
传统方法示意图：
主任务
↓
子任务1 ──→ 子任务2 ──→ 子任务3
↓ ↓ ↓
功能1 功能2 功能3

面向对象软件工程方法：

  • 以对象为核心
  • 封装、继承、多态
  • 更符合人类思维
面向对象方法示意图：
[对象1] ──→ [对象2] ──→ [对象3]
↓ ↓ ↓
属性1 属性2 属性3
方法1 方法2 方法3
↑
封装

★软件生命周期模型

瀑布模型、快速原型模型、增量模型、螺旋模型、喷泉模型

  • 瀑布模型：阶段划分清晰，文档驱动
    • 例子：大型系统开发，如银行核心系统、航空订票系统
    • 类比：建造大楼，先设计图纸，再打地基，再建主体，最后装修
    • 特点：每个阶段必须完成才能进入下一阶段
  • 快速原型模型：适用于需求不明确的系统
    • 例子：用户界面设计、移动应用原型开发
    • 类比：先用粘土做一个简单的模型，让用户看是否满意，再用石材精雕细琢
    • 特点：快速构建可运行的原型，让用户参与反馈
  • 增量模型：逐步构建系统
    • 例子：Office软件开发，先发布Word，再添加Excel、PowerPoint等模块
    • 类比：盖房子先盖一个房间投入使用，再逐步增加其他房间
    • 特点：将系统分解为多个增量，逐个开发和交付
  • 螺旋模型：结合瀑布模型和快速原型模型
    • 例子：大型复杂系统，如航空航天系统、军事软件
    • 类比：像螺旋一样上升，每个循环都包含风险分析和原型开发
    • 特点：强调风险分析，每个阶段都有风险评估
  • 喷泉模型：适用于面向对象开发
    • 例子：面向对象的图形编辑软件、游戏开发
    • 类比：像喷泉的水流，循环往复，各阶段可以重叠和迭代
    • 特点：开发过程具有迭代性和无缝性，各阶段可以并行进行
★软件维护类型（4种）

改正性维护、适应性维护、完善性维护、预防性维护

  • 改正性维护：诊断和改正软件错误（最基础）
    • 例子：修复银行系统中的利息计算错误
    • 例子：修正电商网站购物车计算总价的bug
    • 特点：被动响应，修复已发现的错误
  • 适应性维护：适应环境变化（如操作系统、接口变化）
    • 例子：将旧系统从Windows Server 2003迁移到Windows Server 2019
    • 例子：适配新的API接口，如微信支付接口升级
    • 特点：主动适应环境变化
  • 完善性维护：增加功能或改进性能（占比最多）
    • 例子：在聊天软件中添加语音消息功能
    • 例子：优化数据库查询速度，提升系统响应性能
    • 特点：满足用户新增需求，提升用户体验
  • 预防性维护：为未来可能的改变做准备
    • 例子：重构老旧代码，提高代码可读性和可维护性
    • 例子：添加日志记录功能，便于未来排查问题
    • 特点：前瞻性维护，防止未来可能出现的问题
★软件项目管理概述

软件项目管理的定义：软件项目管理是为了使软件项目能够按照预定的成本、进度、质量顺利完成，而对成本、人员、进度、质量、风险等进行分析和管理的活动。

软件项目的特点：

  • 智力密集型：高度依赖人的创造性思维
  • 劳动密集型：需要大量人力投入
  • 产品无形性：软件是逻辑产品，不可见
  • 开发复杂性：涉及多学科知识的综合应用
  • 过程多变性：需求变更频繁

软件项目管理的重要性：

  • 提高项目成功率
  • 控制成本和进度
  • 保证产品质量
  • 降低项目风险
  • 提高客户满意度
★软件项目组织管理

项目组织结构：

  • 民主制程序员组：
    • 所有成员地位平等，共同决策
    • 适合规模较小、技术水平较高的团队
    • 优点：沟通效率高，成员积极性强
    • 缺点：缺乏统一指挥，决策效率低
  • 主程序员组：
    • 由主程序员、后备程序员、编程秘书组成
    • 主程序员负责主要设计和编码工作
    • 适合技术复杂、质量要求高的项目
  • 层次式程序员组：
    • 设立组长，组员向组长汇报
    • 适合规模较大、层次分明的项目
    • 优点：权责明确，管理有序
    • 缺点：沟通效率可能受影响

团队建设：

  • 选择合适的团队成员
  • 建立良好的团队文化
  • 加强团队沟通与协作
  • 设定明确的团队目标
★软件项目进度管理

进度计划制定：

  • 分解工作任务（WBS - 工作分解结构）
  • 估算工作量和持续时间
  • 确定任务间的依赖关系
  • 分配资源和人员

进度管理工具：

  • 甘特图（Gantt Chart）：
    • 横轴表示时间，纵轴表示任务
    • 用条状图显示任务的起止时间和持续时间
    • 直观反映项目进度状况
  • 网络图（PERT/CPM）：
    • 节点表示事件，箭线表示活动
    • 能够识别关键路径
    • 有助于优化项目时间安排
  • 里程碑（Milestone）：
    • 标志项目重要阶段的完成
    • 便于项目监控和控制
    • 提供项目进展的可见标志
★软件项目成本管理

软件成本构成：

  • 人员费用：占总成本的60%-80%
  • 软硬件设备费用
  • 培训和支持费用
  • 管理费用

成本估算方法：

  • 专家估算法：依靠专家经验进行估算
  • 类比估算法：参考类似项目的成本数据
  • 算法模型法：使用数学模型进行估算
  • 自下而上法：逐项估算后汇总

常用的估算模型：

  • COCOMO模型：构造性成本模型
    • 基本COCOMO：静态单变量模型
    • 中级COCOMO：考虑多个成本驱动因素
    • 详细COCOMO：按软件生命周期各阶段分别估算
  • Putnam模型：动态多变量模型，基于工作量分布曲线
★软件项目质量管理

软件质量的特性：

  • 功能性：满足明确和隐含的需求能力
  • 可靠性：在规定条件下维持性能水平的能力
  • 易用性：用户使用软件的难易程度
  • 效率性：在规定条件下性能水平与所用资源量的关系
  • 可维护性：修改软件的难易程度
  • 可移植性：从一个环境转移到另一个环境的能力

质量管理活动：

  • 质量保证（QA）：确保遵循合适开发标准和规程
  • 质量控制（QC）：验证产品质量是否满足要求
  • 技术评审：检查软件产品的质量
  • 代码走查：检查代码质量和规范性

软件质量标准：

  • ISO 9000系列：国际质量管理体系标准
  • ISO/IEC 25000系列：软件产品质量标准
  • CMM/CMMI：软件能力成熟度模型
★软件项目风险管理

软件项目风险类型：

  • 项目风险：与项目管理和项目环境相关的风险
    • 资金不足、进度延误、人员流失、需求变更等
  • 技术风险：与软件开发技术和工具相关的风险
    • 技术不成熟、性能不达标、接口不兼容等
  • 商业风险：与市场和商业环境相关的风险
    • 市场竞争、法律问题、销售策略等

风险管理过程：

  1. 风险识别：识别潜在的风险来源
  2. 风险评估：分析风险的概率和影响程度
  3. 风险规划：制定风险应对策略
  4. 风险控制：实施风险缓解措施
  5. 风险监控：跟踪风险状态并更新

风险应对策略：

  • 规避：采取行动消除风险产生的条件
  • 转移：将风险的影响转移到第三方
  • 减轻：减少风险发生的概率或影响
  • 接受：接受风险的存在并准备应急计划
