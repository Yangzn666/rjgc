★软件危机

定义：软件开发过程中遇到的各种问题和困难

表现：

  • 软件成本高：如IBM的System/360操作系统项目，预算1000万美元，最终花费超过5000万美元
  • 软件质量差：程序存在大量错误，可靠性差，如早期的银行系统经常出现计算错误
  • 开发进度难以控制：项目延期严重，如原定6个月完成的项目可能拖到2-3年
  • 软件维护困难：代码难以理解，修改容易引入新错误

原因：

  • 需求不明确：用户需求频繁变更，开发过程中不断提出新要求
  • 开发技术落后：缺乏有效的开发方法和工具
  • 管理困难：项目缺乏有效计划和监控机制
  • 缺乏正确的方法指导：没有规范的开发流程

生活化类比：软件危机就像建造房屋时没有设计图纸，工人边建边改，结果工期拖延、成本超支、质量低劣。

★软件工程基本原理

理解其核心思想和实践意义（共7条原理）

软件工程的7条基本原理：

  1. 用分阶段的生命周期计划严格管理
  2. 坚持进行阶段评审
  3. 实行严格的产品控制
  4. 采用现代程序设计技术
  5. 结果应能清楚地审查
  6. 开发小组人员应少而精
  7. 承认不断改进软件工程实践的必要性
★软件工程三要素

方法、工具、过程

  • 方法：软件开发的方法论
    • 例子：结构化分析设计方法、面向对象分析设计方法
    • 类比：就像建筑中的施工方法（钢筋混凝土结构 vs 木结构）
  • 工具：支持软件开发的工具
    • 例子：需求分析工具（如Visio）、编码工具（如IDE）、测试工具（如JUnit）
    • 类比：就像建筑中的工具（锤子、锯子、起重机）
  • 过程：软件开发的步骤和流程
    • 例子：瀑布模型、敏捷开发流程
    • 类比：就像建筑中的施工流程（设计→打地基→建框架→装修）
★软件工程方法

传统软件工程方法 vs 面向对象软件工程方法

传统软件工程方法：

  • 以功能分解为核心
  • 自顶向下设计
  • 结构化分析和设计
传统方法示意图：
主任务
↓
子任务1 ──→ 子任务2 ──→ 子任务3
↓ ↓ ↓
功能1 功能2 功能3

面向对象软件工程方法：

  • 以对象为核心
  • 封装、继承、多态
  • 更符合人类思维
面向对象方法示意图：
[对象1] ──→ [对象2] ──→ [对象3]
↓ ↓ ↓
属性1 属性2 属性3
方法1 方法2 方法3
↑
封装

★软件生命周期模型

瀑布模型、快速原型模型、增量模型、螺旋模型、喷泉模型

  • 瀑布模型：阶段划分清晰，文档驱动
    • 例子：大型系统开发，如银行核心系统、航空订票系统
    • 类比：建造大楼，先设计图纸，再打地基，再建主体，最后装修
    • 特点：每个阶段必须完成才能进入下一阶段
  • 快速原型模型：适用于需求不明确的系统
    • 例子：用户界面设计、移动应用原型开发
    • 类比：先用粘土做一个简单的模型，让用户看是否满意，再用石材精雕细琢
    • 特点：快速构建可运行的原型，让用户参与反馈
  • 增量模型：逐步构建系统
    • 例子：Office软件开发，先发布Word，再添加Excel、PowerPoint等模块
    • 类比：盖房子先盖一个房间投入使用，再逐步增加其他房间
    • 特点：将系统分解为多个增量，逐个开发和交付
  • 螺旋模型：结合瀑布模型和快速原型模型
    • 例子：大型复杂系统，如航空航天系统、军事软件
    • 类比：像螺旋一样上升，每个循环都包含风险分析和原型开发
    • 特点：强调风险分析，每个阶段都有风险评估
  • 喷泉模型：适用于面向对象开发
    • 例子：面向对象的图形编辑软件、游戏开发
    • 类比：像喷泉的水流，循环往复，各阶段可以重叠和迭代
    • 特点：开发过程具有迭代性和无缝性，各阶段可以并行进行
★软件维护类型（4种）

改正性维护、适应性维护、完善性维护、预防性维护

  • 改正性维护：诊断和改正软件错误（最基础）
    • 例子：修复银行系统中的利息计算错误
    • 例子：修正电商网站购物车计算总价的bug
    • 特点：被动响应，修复已发现的错误
  • 适应性维护：适应环境变化（如操作系统、接口变化）
    • 例子：将旧系统从Windows Server 2003迁移到Windows Server 2019
    • 例子：适配新的API接口，如微信支付接口升级
    • 特点：主动适应环境变化
  • 完善性维护：增加功能或改进性能（占比最多）
    • 例子：在聊天软件中添加语音消息功能
    • 例子：优化数据库查询速度，提升系统响应性能
    • 特点：满足用户新增需求，提升用户体验
  • 预防性维护：为未来可能的改变做准备
    • 例子：重构老旧代码，提高代码可读性和可维护性
    • 例子：添加日志记录功能，便于未来排查问题
    • 特点：前瞻性维护，防止未来可能出现的问题
★面向对象的基本概念

对象(Object)：系统中用来描述客观事物的一个实体，是构成系统的基本单位。

  • 对象由一组属性和对这组属性进行操作的一组服务构成
  • 对象 = 数据 + 算法
  • 属性：描述对象特征的数据项
  • 服务：对数据进行的操作

类(Class)：具有相同属性和服务的一组对象的集合，是一个抽象的概念。

  • 类是对一系列相似对象的抽象描述
  • 对象是类的具体实例
  • 类是对象的模板，对象是类的实例

封装(Encapsulation)：把对象的属性和服务结合成一个独立的系统单位，并尽可能隐蔽对象的内部细节。

  • 信息隐蔽是通过对象的封装性来实现的
  • 对外提供访问接口，内部实现细节对外不可见
  • 提高了系统的安全性和可维护性

继承(Inheritance)：子类可以获得父类的属性和服务，也可以定义自己的属性和方法。

  • 单继承：一个子类只有一个父类
  • 多继承：一个子类可以有多个父类
  • 提高了软件模块的可重用性和可扩充性

多态(Polymorphism)：同一个操作作用于不同类的对象可以有不同的解释，产生不同的执行结果。

  • 允许不同类的对象对同一消息做出响应
  • 灵活性和可扩展性的体现
  • 方法重载和方法重写都是多态的体现
★面向对象分析(OOA)

面向对象分析的任务：运用面向对象方法进行需求分析，建立系统分析模型。

OOA的主要活动：

  1. 识别对象，进行抽象分类
  2. 标识对象间的静态联系
  3. 定义对象的服务，标识对象间的动态联系
  4. 确定对象的操作方法
  5. 确定约束，给出说明
  6. 整理和修改模型

OOA模型：

  • 对象模型：描述系统的静态结构
  • 动态模型：描述系统的交互序列
  • 功能模型：描述系统的数据变换功能

OOA的特点：

  • 从现实世界中的实体出发认识问题
  • 建立的分析模型易于理解和交流
  • 分析模型对需求的变化具有较强的适应性
★面向对象设计(OOD)

面向对象设计的任务：在OOA模型的基础上，对分析模型进行补充和完善，形成设计模型。

OOD的主要内容：

  • 问题域部分设计
  • 人机交互部分设计
  • 任务管理部分设计
  • 数据管理部分设计

OOD设计准则：

  • 抽象：抽取事物的本质特征
  • 信息隐蔽：对象的实现细节对外部是隐蔽的
  • 功能独立：对象的内部关联要尽量紧密
  • 一致性：设计应保持内在的一致性
  • 完备性：设计应能完全表达问题空间的需求

OOD设计风格：

  • 对象类粒度适中
  • 继承层次适当
  • 消息关联尽量简单
  • 对象间耦合程度要低
★面向对象编程(OOP)

面向对象编程的特点：

  • 封装性：将数据和操作数据的方法绑定在一起
  • 继承性：子类可以继承父类的属性和方法
  • 多态性：同一个接口可以有不同的实现

常用面向对象编程语言：

  • C++：在C语言基础上增加了面向对象特性
  • Java：纯面向对象编程语言，跨平台性强
  • C#：微软开发的面向对象语言
  • Python：支持多种编程范式，包括面向对象

OOP的优点：

  • 可重用性高
  • 可维护性好
  • 可扩展性强
  • 代码可读性好
★UML统一建模语言

UML概述：Unified Modeling Language，是一种通用的可视化建模语言，用于软件密集型系统的分析、设计和构造。

UML图的分类：

  • 结构图：描述系统的静态结构
    • 类图：描述类、接口以及它们之间的关系
    • 对象图：表示类的对象实例
    • 组件图：描述组件及其依赖关系
    • 部署图：描述系统中硬件和软件的物理配置
  • 行为图：描述系统的动态行为
    • 用例图：描述系统功能及参与者与系统的关系
    • 活动图：描述业务流程或操作的工作流程
    • 状态图：描述对象的状态转换
    • 序列图：描述对象间的交互序列

UML的优势：

  • 标准化的建模语言
  • 可视化程度高
  • 与具体实现无关
  • 支持面向对象的分析、设计和实现
★面向对象设计原则

六大设计原则：

  1. 单一职责原则(SRP)：一个类应该仅有一个引起它变化的原因
  2. 开闭原则(OCP)：对扩展开放，对修改封闭
  3. 里氏替换原则(LSP)：子类型必须能够替换掉它们的基类型
  4. 依赖倒置原则(DIP)：高层模块不应该依赖底层模块，两者都应该依赖于抽象
  5. 接口隔离原则(ISP)：客户端不应该依赖不需要的接口
  6. 迪米特法则(LoD)：一个对象应当对其他对象有尽可能少的了解

设计模式分类：

  • 创建型模式：工厂模式、抽象工厂模式、单例模式、建造者模式、原型模式
  • 结构型模式：适配器模式、桥接模式、组合模式、装饰器模式、外观模式、享元模式、代理模式
  • 行为型模式：责任链模式、命令模式、解释器模式、迭代器模式、中介者模式、备忘录模式、观察者模式、状态模式、策略模式、模板方法模式、访问者模式
