简答题（共5题）

★1. 什么是软件危机？其表现有哪些？
软件危机是指在计算机软件开发和维护过程中所遇到的一系列严重问题。其主要表现包括：

1. 软件成本高：软件开发费用超出预算，开发成本持续增长
2. 软件质量差：软件存在大量错误，可靠性差
3. 开发进度难以控制：项目延期严重，无法按时交付
4. 维护困难：软件维护成本高，难以修改和升级

例如：IBM的System/360操作系统项目，预算1000万美元，最终花费超过5000万美元。
★2. 什么是模块独立性？如何衡量模块独立性？
模块独立性是指软件系统中每个模块只完成系统要求的独立的子功能，并且与其他模块之间的接口简单。

模块独立性可以通过两个指标来衡量：
1. 耦合：衡量不同模块彼此间相互依赖的紧密程度。目标是低耦合，模块间联系尽可能松散
2. 内聚：衡量一个模块内部各元素彼此结合的紧密程度。目标是高内聚，模块内部功能高度相关

模块独立性的好处：提高模块的可维护性和可重用性，降低系统复杂性。
★3. 白盒测试和黑盒测试有什么区别？
白盒测试和黑盒测试的主要区别：

1. 白盒测试（结构测试）：
- 基于程序内部逻辑结构进行测试
- 测试者了解程序内部实现
- 主要方法：逻辑覆盖、基本路径测试
- 目标：确保程序内部逻辑路径正确

2. 黑盒测试（功能测试）：
- 基于程序功能和外部行为进行测试
- 测试者不了解程序内部实现
- 主要方法：等价类划分、边界值分析、因果图法
- 目标：验证程序是否满足功能需求

两种方法互补使用，通常在单元测试阶段多用白盒测试，在系统测试阶段多用黑盒测试。
★4. 什么是软件维护？四种维护类型分别是什么？
软件维护是指软件系统交付使用后，为了改正错误或满足新的需要而修改软件的过程。

四种维护类型：
1. 改正性维护：诊断和改正软件中潜藏的错误
2. 适应性维护：使软件适应环境的变化
3. 完善性维护：增加软件功能或改善软件性能
4. 预防性维护：为了提高软件的可维护性和可靠性而进行的修改

其中，完善性维护占比最大，约占软件维护工作量的50%以上。
★5. 面向对象方法学的四个要点是什么？
面向对象方法学的四个要点：

1. 认为对象是基本的运行时实体：现实世界的任何事物都可以看作对象，对象既有属性又有操作
2. 强调对象间的消息传递：对象之间通过消息进行通信和协作
3. 把所有对象都划分成各种对象类：相同属性和操作的对象归为一类，类是对象的抽象
4. 对具有相同特性的对象抽象成类：通过继承机制实现代码复用和系统扩展

面向对象方法学的特点：与人类思维方式一致、稳定性好、可重用性强、易于开发大型软件和维护。

综合题（共3题）

★1. 比较传统软件工程方法和面向对象软件工程方法的优缺点，并说明各自的适用场景。
传统软件工程方法：
优点：结构清晰，开发过程规范，适合功能明确的系统。例如：早期的批处理系统、操作系统等。
缺点：对需求变化适应性差，难以应对复杂系统。例如：需求频繁变更的电商平台开发困难。

面向对象软件工程方法：
优点：与人类思维一致，稳定性好，可重用性强，易于开发大型软件和维护。例如：Java企业级应用、现代Web框架等。
缺点：学习成本较高，对设计人员要求更高。例如：需要深入理解封装、继承、多态等概念。

适用场景：
- 传统方法：适用于需求明确且稳定的系统，如系统工具、嵌入式系统等
- 面向对象方法：适用于需求变化较多、规模较大的复杂系统，如企业级应用、图形界面应用等
★2. 详细说明软件测试的各个步骤及其主要目的，并分析各步骤之间的关系。
软件测试的四个主要步骤：

1. 单元测试：
- 目的：测试单个模块，确保每个模块功能正确
- 方法：主要使用白盒测试方法，检查模块内部逻辑
- 依据：模块详细设计说明书

2. 集成测试：
- 目的：测试模块间接口和交互，确保模块组合后功能正确
- 方法：渐增式集成或非渐增式集成，自顶向下或自底向上
- 依据：软件概要设计说明书

3. 确认测试：
- 目的：验证软件是否满足需求，确保软件符合用户要求
- 方法：主要使用黑盒测试方法，验证功能和性能
- 依据：软件需求规格说明书

4. 系统测试：
- 目的：测试整个系统，确保系统在实际环境中正常运行
- 方法：在真实环境下进行全面测试
- 依据：系统需求和用户环境

各步骤关系：是一个逐步深化的过程，前一步骤的输出是后一步骤的输入。只有当单元测试通过后才能进行集成测试，依次类推。
★3. 分析瀑布模型、快速原型模型、增量模型、螺旋模型的优缺点及适用场景，并说明它们之间的关系。
1. 瀑布模型：
- 优点：阶段划分清晰，文档驱动，便于管理控制
- 缺点：缺乏灵活性，难以适应需求变更，用户反馈较晚
- 适用：需求明确且稳定的项目，如大型系统开发

2. 快速原型模型：
- 优点：用户可及早参与反馈，需求理解更准确
- 缺点：可能忽略质量与性能要求，文档可能不完整
- 适用：需求不明确的系统，如用户界面设计

3. 增量模型：
- 优点：可分阶段交付产品，降低风险，便于管理
- 缺点：需要良好的体系结构设计
- 适用：大型项目分阶段交付，如Office软件开发

4. 螺旋模型：
- 优点：结合瀑布模型和快速原型模型，强调风险分析
- 缺点：复杂度高，成本较高，需要风险评估经验
- 适用：大型复杂系统，如航空航天系统

它们之间的关系：都是软件开发过程模型，各有特点。瀑布模型是最基础的模型，其他模型是对它的改进和扩展。螺旋模型综合了其他模型的优点，但更为复杂。选择模型时需要根据项目特点和风险因素来决定。
