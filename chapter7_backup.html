定义：软件开发过程中遇到的各种问题和困难

表现：

  • 软件成本高：如IBM的System/360操作系统项目，预算1000万美元，最终花费超过5000万美元
  • 软件质量差：程序存在大量错误，可靠性差，如早期的银行系统经常出现计算错误
  • 开发进度难以控制：项目延期严重，如原定6个月完成的项目可能拖到2-3年
  • 软件维护困难：代码难以理解，修改容易引入新错误

原因：

  • 需求不明确：用户需求频繁变更，开发过程中不断提出新要求
  • 开发技术落后：缺乏有效的开发方法和工具
  • 管理困难：项目缺乏有效计划和监控机制
  • 缺乏正确的方法指导：没有规范的开发流程

生活化类比：软件危机就像建造房屋时没有设计图纸，工人边建边改，结果工期拖延、成本超支、质量低劣。

理解其核心思想和实践意义（共7条原理）

软件工程的7条基本原理：

  1. 用分阶段的生命周期计划严格管理
  2. 坚持进行阶段评审
  3. 实行严格的产品控制
  4. 采用现代程序设计技术
  5. 结果应能清楚地审查
  6. 开发小组人员应少而精
  7. 承认不断改进软件工程实践的必要性

方法、工具、过程

  • 方法：软件开发的方法论
    • 例子：结构化分析设计方法、面向对象分析设计方法
    • 类比：就像建筑中的施工方法（钢筋混凝土结构 vs 木结构）
  • 工具：支持软件开发的工具
    • 例子：需求分析工具（如Visio）、编码工具（如IDE）、测试工具（如JUnit）
    • 类比：就像建筑中的工具（锤子、锯子、起重机）
  • 过程：软件开发的步骤和流程
    • 例子：瀑布模型、敏捷开发流程
    • 类比：就像建筑中的施工流程（设计→打地基→建框架→装修）

传统软件工程方法 vs 面向对象软件工程方法

传统软件工程方法：

  • 以功能分解为核心
  • 自顶向下设计
  • 结构化分析和设计
传统方法示意图：
主任务
↓
子任务1 ──→ 子任务2 ──→ 子任务3
↓ ↓ ↓
功能1 功能2 功能3

面向对象软件工程方法：

  • 以对象为核心
  • 封装、继承、多态
  • 更符合人类思维
面向对象方法示意图：
[对象1] ──→ [对象2] ──→ [对象3]
↓ ↓ ↓
属性1 属性2 属性3
方法1 方法2 方法3
↑
封装
★软件生命周期

三大阶段：定义阶段、开发阶段、维护阶段

具体阶段：

  1. 可行性研究
  2. 需求分析
  3. 总体设计
  4. 详细设计
  5. 编码实现
  6. 综合测试
  7. 维护

瀑布模型、快速原型模型、增量模型、螺旋模型、喷泉模型

  • 瀑布模型：阶段划分清晰，文档驱动
    • 例子：大型系统开发，如银行核心系统、航空订票系统
    • 类比：建造大楼，先设计图纸，再打地基，再建主体，最后装修
    • 特点：每个阶段必须完成才能进入下一阶段
  • 快速原型模型：适用于需求不明确的系统
    • 例子：用户界面设计、移动应用原型开发
    • 类比：先用粘土做一个简单的模型，让用户看是否满意，再用石材精雕细琢
    • 特点：快速构建可运行的原型，让用户参与反馈
  • 增量模型：逐步构建系统
    • 例子：Office软件开发，先发布Word，再添加Excel、PowerPoint等模块
    • 类比：盖房子先盖一个房间投入使用，再逐步增加其他房间
    • 特点：将系统分解为多个增量，逐个开发和交付
  • 螺旋模型：结合瀑布模型和快速原型模型
    • 例子：大型复杂系统，如航空航天系统、军事软件
    • 类比：像螺旋一样上升，每个循环都包含风险分析和原型开发
    • 特点：强调风险分析，每个阶段都有风险评估
  • 喷泉模型：适用于面向对象开发
    • 例子：面向对象的图形编辑软件、游戏开发
    • 类比：像喷泉的水流，循环往复，各阶段可以重叠和迭代
    • 特点：开发过程具有迭代性和无缝性，各阶段可以并行进行

改正性维护、适应性维护、完善性维护、预防性维护

  • 改正性维护：诊断和改正软件错误（最基础）
    • 例子：修复银行系统中的利息计算错误
    • 例子：修正电商网站购物车计算总价的bug
    • 特点：被动响应，修复已发现的错误
  • 适应性维护：适应环境变化（如操作系统、接口变化）
    • 例子：将旧系统从Windows Server 2003迁移到Windows Server 2019
    • 例子：适配新的API接口，如微信支付接口升级
    • 特点：主动适应环境变化
  • 完善性维护：增加功能或改进性能（占比最多）
    • 例子：在聊天软件中添加语音消息功能
    • 例子：优化数据库查询速度，提升系统响应性能
    • 特点：满足用户新增需求，提升用户体验
  • 预防性维护：为未来可能的改变做准备
    • 例子：重构老旧代码，提高代码可读性和可维护性
    • 例子：添加日志记录功能，便于未来排查问题
    • 特点：前瞻性维护，防止未来可能出现的问题
★软件测试的目标和原则

软件测试的目标：

  • 发现软件中的错误和缺陷
  • 验证软件是否满足需求规格说明书的要求
  • 评估软件的质量和可靠性
  • 确保软件能够在预定环境下正常运行

软件测试的原则：

  • 尽早开始测试：越早发现问题，修复成本越低
  • 测试用例必须明确预期结果
  • 测试应基于客户需求和规格说明
  • 测试无法证明软件无错，只能证明有错
  • Pareto原则：80%的错误可能来自20%的程序模块
★测试方法分类

按测试技术分类：

  • 白盒测试：也称结构测试或逻辑驱动测试，基于程序内部结构进行测试
    • 测试覆盖标准：语句覆盖、分支覆盖、条件覆盖、路径覆盖等
    • 主要关注程序内部逻辑结构
    • 常用于单元测试阶段
  • 黑盒测试：也称功能测试或数据驱动测试，只关心输入输出
    • 不考虑程序内部结构，只根据功能需求进行测试
    • 主要用于集成测试和系统测试阶段
    • 测试用例基于需求规格说明书设计

按测试执行阶段分类：

  • 静态测试：不执行程序，通过阅读代码、文档等方式发现错误
  • 动态测试：通过运行程序来检验程序行为和结果
★软件测试的各个阶段

四个主要测试阶段：

  1. 单元测试（模块测试）：
    • 针对单个模块进行测试
    • 通常由程序员自己完成
    • 主要使用白盒测试技术
    • 测试内容：模块接口、局部数据结构、重要的执行路径、边界条件、出错处理
  2. 集成测试（组装测试）：
    • 将模块组装成系统的过程
    • 两种策略：非增量集成和增量集成
    • 增量集成方式：自顶向下、自底向上、混合策略
  3. 确认测试（验收测试）：
    • 验证软件是否满足需求规格说明书的要求
    • 以用户为主导，开发人员协助
    • Alpha测试和Beta测试
  4. 系统测试：
    • 将通过确认测试的软件与计算机硬件、外设、支撑软件等组合在一起进行测试
    • 包括：恢复测试、安全性测试、强度测试、性能测试、可靠性测试等
★测试用例设计技术

黑盒测试用例设计技术：

  • 等价类划分：
    • 将输入域划分为若干等价类
    • 从每个等价类中选取一个代表值进行测试
    • 有效等价类和无效等价类
  • 边界值分析：
    • 在等价类边界及其相邻区域进行测试
    • 大量错误发生在输入范围的边界附近
    • 测试边界值和边界附近的值
  • 因果图法：
    • 利用图解法分析输入条件的各种组合情况
    • 适合于检查输入条件之间的相互制约关系
    • 将因果图转换为决策表，再设计测试用例
  • 错误推测法：
    • 基于经验和直觉推测程序中可能存在的各种错误
    • 有针对性地设计测试用例
    • 常与其他测试方法结合使用
★调试（排错）

调试是在测试发现错误之后排除错误的过程。

调试的步骤：

  1. 从错误征象确定可能的出错位置
  2. 从错误征象追溯到错误根源
  3. 从错误根源消除错误

常用的调试方法：

  • 蛮力法：通过计算机显示大量调试信息来排除错误
  • 回溯法：沿着程序逻辑反向追踪源代码错误
  • 原因排除法：通过对错误征象进行分析，列出可能的原因，逐一排除

测试与调试的区别：

  • 测试是为了发现问题，调试是为了定位和修复问题
  • 测试可以由独立的测试团队完成，调试通常由开发人员完成
★软件测试策略

测试策略的制定原则：

  • 在有限的时间和资源下进行最有效的测试
  • 保证测试的全面性和针对性
  • 优先测试最容易发生错误的部分

测试策略的关键要素：

  • 测试充分性：确定何时停止测试
  • 测试覆盖标准：衡量测试的完整性
  • 测试数据生成技术：如何生成有效的测试数据
  • 测试评价标准：如何评估测试结果
  • 测试管理技术：测试过程的组织和管理
